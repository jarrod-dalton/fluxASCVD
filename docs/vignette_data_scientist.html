Implementing an Event-Driven ASCVD Simulator with patientSimCore

Overview

This vignette is for data scientists implementing a pedagogical ASCVD care trajectory model on top of patientSimCore. The objective is to teach the mechanics of building a clinically credible event-driven model: jittered clinic visits with no-shows, encounter-driven labs (ordered at visits and drawn later), ordinal treatment escalation, and a terminal composite ASCVD event process.

The statistical models here are deliberately simple placeholders. The key architectural point is that you can replace the placeholders with ML models without changing the Engine–ModelBundle contract.

This package (patientSimASCVD) depends on patientSimCore (>= 0.10.0).

How the pieces fit

At runtime:

  1. A Patient exists with event-sourced state and a global time axis (years in this model).
  2. An Engine asks the bundle for proposed future events across processes.
  3. The earliest event is selected, and transition() is applied.
  4. Some processes refresh their proposals (default: after any state update).
  5. The simulation stops when stop() returns TRUE (here: a terminal ASCVD event).

The package entry points

The model package provides:

  • ascvd_schema() (model-specific schema, including blocks)
  • ascvd_model_bundle() (a ModelBundle compatible with patientSimCore::Engine)

Below we walk through the actual code in this repo and explain the intent of each part.

Schema: state, blocks, and required variables

The schema defines what state exists, defaults, and how variables are grouped into clinical panels (blocks).

In the pedagogical model we group: - BP: sbp, dbp - BMP: sodium, potassium, creatinine, glucose - Lipids: LDL, HDL, triglycerides - Treatments: antihypertensive count and statin intensity - Operational ordering state: order times for BMP and lipids

Here is the current schema implementation:

# ------------------------------------------------------------------------------
# ASCVD patient schema
#
# Defines required core state, panels (blocks), treatments, and lab order state.
# Age and sex are required at patient instantiation.
# ------------------------------------------------------------------------------
ascvd_schema <- function() {
  schema <- patientSimCore::default_patient_schema()

  schema$age$required <- TRUE
  schema$sex$required <- TRUE

  schema$sbp$blocks <- "bp"
  schema$dbp$blocks <- "bp"

  schema$sodium$blocks <- "bmp"
  schema$potassium$blocks <- "bmp"
  schema$creatinine$blocks <- "bmp"
  schema$glucose$blocks <- "bmp"

  schema$ldl$blocks <- "lipids"
  schema$hdl$blocks <- "lipids"
  schema$triglycerides$blocks <- "lipids"

  schema$n_antihypertensives <- list(default = 0, blocks = "tx_htn")
  schema$statin_intensity <- list(default = "none", blocks = "tx_lipid")

  schema$bmp_order_time <- list(default = NA_real_)
  schema$lipid_order_time <- list(default = NA_real_)

  schema$ascvd <- list(default = 0)

  schema
}

What to change later - Mark age and sex as required (preferred) and enforce at patient instantiation. - Add explicit domains (ordered factor for statins; integer 0–4 for antihypertensives). - Define blocks as many-to-many if you later add overlap (e.g., CMP containing BMP).

ModelBundle constructor

patientSimCore expects a named list with functions implementing the ModelBundle contract.

Here is the bundle constructor:

# ------------------------------------------------------------------------------
# ASCVD ModelBundle constructor
# ------------------------------------------------------------------------------
ascvd_model_bundle <- function(params = list()) {
  list(
    propose_events = propose_events_ascvd,
    transition = transition_ascvd,
    stop = stop_ascvd
  )
}

In a fuller implementation, this is where you would: - accept a params list (or draws) and close over them - attach observe() if you want an observation table - supply custom refresh_rules if you want process-specific refresh behavior

Proposing events: multiple processes on one time axis

propose_events(patient, ctx) returns a list of candidate events, typically one per process. Each candidate includes a time_next, an event_type, and a process_id.

This repo uses processes: - clinic: proposes the next scheduled clinic visit - bmp: proposes a BMP draw only if ordered - lipids: proposes a lipid panel draw only if ordered - ascvd: proposes a terminal ASCVD event time

Here is the proposal function:

propose_events_ascvd <- function(patient, ctx) {
  events <- list()

  # Clinic visit proposal
  t_next <- patient$time + stats::rgamma(1, shape = 4, rate = 12)
  events$clinic <- list(
    time_next = t_next,
    event_type = "clinic_visit",
    process_id = "clinic"
  )

  # BMP draw if ordered
  bmp_time <- patient$state("bmp_order_time")
  if (!is.na(bmp_time)) {
    events$bmp <- list(
      time_next = bmp_time,
      event_type = "bmp_draw",
      process_id = "bmp"
    )
  }

  # Lipid draw if ordered
  lipid_time <- patient$state("lipid_order_time")
  if (!is.na(lipid_time)) {
    events$lipids <- list(
      time_next = lipid_time,
      event_type = "lipid_draw",
      process_id = "lipids"
    )
  }

  # Terminal ASCVD event
  lambda <- 0.02
  t_ascvd <- patient$time + stats::rexp(1, rate = lambda)
  events$ascvd <- list(
    time_next = t_ascvd,
    event_type = "ascvd_event",
    process_id = "ascvd",
    ascvd_type = sample(c("mi", "stroke", "death"), 1)
  )

  events
}

Encounter-driven labs (Option B)

Notice the lab processes only propose events if bmp_order_time or lipid_order_time is set. This is the core mechanism that makes labs encounter-driven:

  • At a clinic visit, transition() can place orders by setting bmp_order_time / lipid_order_time.
  • The proposal layer sees the order and schedules the draw.
  • The lab draw transition clears the order back to NA.

That is a clinically interpretable “order → draw → result” lifecycle, and it is easy to extend with turnaround delays or noncompliance.

Transitions: atomic state updates per event

transition(patient, event, ctx) receives the full event object. It returns either: - a named list of state changes (applied as an atomic update), or - NULL (meaning: no state changes for this event)

Here is the current transition implementation:

transition_ascvd <- function(patient, event, ctx) {
  et <- event$event_type

  if (et == "clinic_visit") {
    # No-show model (simple)
    if (stats::runif(1) < 0.2) return(NULL)

    updates <- list()

    # BP update
    sbp <- patient$state("sbp") + stats::rnorm(1, -5, 10)
    dbp <- patient$state("dbp") + stats::rnorm(1, -3, 5)
    updates <- c(updates, list(sbp = sbp, dbp = dbp))

    # HTN intensification
    if (sbp > 130 || dbp > 80) {
      n <- patient$state("n_antihypertensives")
      updates$n_antihypertensives <- min(n + 1, 4)
    }

    # Order labs
    updates$bmp_order_time <- event$time_next
    updates$lipid_order_time <- event$time_next

    return(updates)
  }

  if (et == "bmp_draw") {
    return(list(
      sodium = stats::rnorm(1, 140, 2),
      potassium = stats::rnorm(1, 4.2, 0.3),
      creatinine = stats::rnorm(1, 1.0, 0.1),
      glucose = stats::rnorm(1, 100, 10),
      bmp_order_time = NA_real_
    ))
  }

  if (et == "lipid_draw") {
    return(list(
      ldl = stats::rnorm(1, 100, 20),
      hdl = stats::rnorm(1, 50, 10),
      triglycerides = stats::rnorm(1, 150, 40),
      lipid_order_time = NA_real_
    ))
  }

  if (et == "ascvd_event") {
    return(list(ascvd = 1))
  }

  NULL
}

Clinic visits and no-shows

In this pedagogical code, no-shows are implemented as a random branch. When a no-show occurs, transition() returns NULL.

That means: - no BP measurement - no treatment changes - no lab orders - no state changes at all

The missed visit is still an event in the log; derived variables (like a no-show count) can be computed from history.

BP as a vectorized update

SBP/DBP are updated together (even in this simple implementation). In a fuller version, you would compute a joint update (e.g., multivariate normal) and then apply it as a BP block update using patientSimCore::update_block() and patientSimCore::combine_updates().

Treatment escalation

The current code intensifies antihypertensives when SBP > 130 or DBP > 80, capped at 4. This is a placeholder for a decision model. Later you can: - fit an ordinal model for escalation probability (inertia) - replace with an ML policy model - incorporate contraindications or adverse labs

Ordering labs at visits

When a visit is attended, this code places both BMP and lipid orders immediately by setting order times to the visit time. In a more realistic version you would condition orders on “due-ness” (time since last draw), risk tier, recent abnormalities, or medication changes.

Terminal ASCVD events and stopping

The ASCVD process proposes an event time. When that event occurs, transition() sets a terminal indicator and stop() halts the simulation.

Here is the stop function:

stop_ascvd <- function(patient, event, ctx) {
  if (event$event_type == "ascvd_event") return(TRUE)
  FALSE
}

You will typically also include a time-horizon stop (e.g., 10 years) for cohort simulations, even if a terminal event might occur earlier.

Running a single patient

A typical minimal run looks like:

# Schema + patient
schema <- ascvd_schema()
p <- patientSimCore::new_patient(
  init = list(age = 60, sex = "M"),
  schema = schema
)

# Bundle + engine
bundle <- ascvd_model_bundle()
eng <- patientSimCore::Engine$new(bundle = bundle)

# Run
out <- eng$run(
  patient = p,
  ctx = list(time_unit = "years"),
  max_events = 50,
  return_observations = FALSE
)

# Inspect patient event log and final state
out$patient$events()
out$patient$state()

Where to swap in ML models (without changing the contract)

This pedagogical model is intentionally modular. The best swap-in points are pure functions that return:

  • a visit interval distribution (or direct next-visit time)
  • a no-show probability
  • a joint BP update (SBP/DBP)
  • a BMP panel update
  • a lipid panel update
  • treatment decisions (ordinal escalation)
  • a terminal ASCVD hazard or survival distribution

As long as these functions return the same types expected by the transition and proposal layers, the Engine-facing ModelBundle contract does not change.

Closing note

The engine is not the model. The model is the explicit set of assumptions about events, measurements, decisions, and risk. The point of patientSimASCVD is to make those assumptions visible today, and swappable tomorrow.
